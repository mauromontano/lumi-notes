# Lumi Notes v1.1 — Diseño de 5 features

**Fecha:** 2026-07-04
**Estado:** aprobado en brainstorming (pendiente de plan de implementación)

## Contexto

Lumi Notes v1 (Expo 57, iOS-first, dev build) ya tiene: notas por texto y voz con formateo por Claude (API key opcional), recordatorios con recurrencia (expo-notifications), SQLite (expo-sqlite), temas dark/light, y deep link `luminotes://`. Este ciclo agrega 5 features en fases ordenadas de menor a mayor complejidad. Cada fase se implementa, prueba y puede shippearse por separado.

## Decisiones de producto (validadas con el usuario)

1. **Notificaciones**: acciones "Posponer 1h" y "Completar". Completar limpia el recordatorio puntual; en recurrentes solo descarta la ocurrencia. La nota nunca se borra.
2. **Tags IA**: set fijo de categorías (`compras`, `trabajo`, `ideas`, `personal`, `salud`, `viajes`), 1 tag por nota, editable a mano. Sin API key ⇒ sin tag.
3. **Share extension**: compartir texto abre Lumi en el preview con el texto formateado por IA (crudo sin key). Lib `expo-share-intent`.
4. **Siri**: App Shortcut "Dictar nota en Lumi" que abre `luminotes://voice` vía App Intent en el target principal. Sin Control Widget en esta versión.
5. **Notas cifradas**: cuerpo en Keychain de iOS con Face ID obligatorio (expo-secure-store + `requireAuthentication`). Pensadas para keys/credenciales (límite ~2KB). Nunca pasan por la IA.

## Fase 1 — Acciones en notificaciones de recordatorio

**Módulos:** `src/reminders/actions.ts` (nuevo, lógica pura), `src/reminders/scheduler.ts`, `src/app/_layout.tsx`.

- Registrar categoría `reminder` con `Notifications.setNotificationCategoryAsync('reminder', [...])` al iniciar la app: acciones `snooze-1h` ("Posponer 1h") y `complete` ("Completar"), ambas sin abrir la app al foreground si la plataforma lo permite.
- `scheduleReminder` pasa a incluir `categoryIdentifier: 'reminder'` en el content.
- `actions.ts` expone `handleReminderAction(note, actionIdentifier, now)` que decide (función pura, testeable):
  - `snooze-1h`: agendar notificación puntual a `now + 1h` con el mismo contenido → actualizar `notificationId` de la nota. La recurrencia original no se modifica (iOS mantiene el schedule recurrente).
  - `complete`: recordatorio puntual ⇒ limpiar `reminderAt`, `reminderRecurrence`, `notificationId` en la nota; recurrente ⇒ no tocar nada (solo se descarta la ocurrencia).
  - default (tap en el cuerpo): abrir la nota (comportamiento actual).
- En `_layout.tsx`, el listener existente de respuestas rutea por `response.actionIdentifier`. Al iniciar la app, procesar `getLastNotificationResponseAsync()` para cubrir respuestas recibidas con la app cerrada (deduplicar por identifier ya procesado).

**Errores:** si el reagendado falla (permisos revocados), la nota queda como estaba; log `[lumi]`.

## Fase 2 — Tags automáticos por IA + filtro

**Módulos:** `src/notes/tags.ts` (nuevo), `src/db/schema.ts`, `src/db/notesRepo.ts`, `src/ai/formatter.ts` + `claudeFormatter.ts`, `src/app/index.tsx`, `src/components/NoteCard.tsx`, `src/app/note/[id].tsx`.

- `tags.ts`: `export const NOTE_TAGS = ['compras','trabajo','ideas','personal','salud','viajes'] as const;` + color por tag para ambas paletas + `isNoteTag(x)`.
- Migración: `ALTER TABLE notes ADD COLUMN tag TEXT` (nullable). Patrón de migración aditiva sobre `MIGRATION_SQL` existente (usar `user_version` o try/catch de columna duplicada, según el patrón actual de `database.ts`).
- Prompt del formatter: respuesta pasa a `{"titulo","cuerpo","tag"}` donde `tag` es uno del set o `null`. `parseFormatterResponse` valida con `isNoteTag`; tag inválido ⇒ `null` (nunca falla el guardado por el tag).
- Aplica a: dictado nuevo, edición por voz y texto compartido (fase 3). Notas de texto manuales: sin tag automático, editable a mano.
- UI:
  - Home: fila de chips de filtro (horizontal, debajo del SearchBar). Un tag activo a la vez; tocarlo de nuevo lo desactiva. Filtro se combina con la búsqueda (`listNotes` acepta `tag` opcional).
  - `NoteCard`: badge con el tag y su color.
  - Pantalla de nota: selector de chips (los 6 tags + "sin tag") para editar a mano.

## Fase 3 — Compartir texto hacia Lumi (share sheet)

**Módulos:** `app.json` (plugin), `src/app/_layout.tsx` o hook dedicado, `src/app/voice.tsx`.

- Lib `expo-share-intent` (config plugin oficial de la lib; requiere `prebuild` + rebuild del dev client). Habilitar solo texto/URLs en iOS.
- Al recibir un share intent con texto: navegar a `/voice` con parámetro `sharedText`.
- `voice.tsx`: si llega `sharedText`, saltea el dictado — no pide micrófono ni arranca `useDictation` — y entra directo en fase `thinking` → `formatter.formatNote(sharedText)` → `preview` (mismo flujo, mismo orbe, mismos fallbacks sin key/sin red que hoy). Extraer la transición dictado→preview a una función compartida si hace falta.
- El texto compartido respeta el límite razonable (truncar a ~8k chars con aviso).

## Fase 4 — App Shortcut de Siri

**Módulos:** `plugins/withAppIntents.ts` (config plugin local, nuevo), `ios-assets/DictarNotaIntent.swift` (nuevo), `app.json`.

- `DictarNotaIntent.swift`: `AppIntent` con `openAppWhenRun = true` cuyo `perform()` abre `luminotes://voice`; `AppShortcutsProvider` con frases en español e inglés ("Dictar nota en ${applicationName}", "New voice note in ${applicationName}").
- Config plugin local que copia el .swift al target principal en prebuild (withXcodeProject), para sobrevivir `npx expo prebuild --clean`. Sin extensión ni App Group.
- Resultado: funciona "Oye Siri, dictar nota en Lumi", aparece en Spotlight y en la app Atajos (el usuario puede sumarlo al lock screen manualmente).
- Antes de implementar, verificar API exacta de App Intents en la doc de Expo v57 / Apple (regla del repo: leer docs versionadas).

## Fase 5 — Notas cifradas (Keychain + Face ID)

**Módulos:** `src/db/schema.ts`, `src/db/notesRepo.ts`, `src/notes/secureBody.ts` (nuevo wrapper de SecureStore), `src/app/note/[id].tsx`, `src/components/NoteCard.tsx`, `app.json` (`NSFaceIDUsageDescription`).

- Migración: `ALTER TABLE notes ADD COLUMN secure INTEGER NOT NULL DEFAULT 0`.
- `secureBody.ts`: `saveSecureBody(noteId, body)`, `readSecureBody(noteId)`, `deleteSecureBody(noteId)` sobre `expo-secure-store` con clave `note-body-<id>` y `requireAuthentication: true` (Face ID/código lo exige el Keychain al leer). Límite: si `body` supera ~2000 bytes, error tipado `secure-body-too-long` → aviso en UI antes de guardar.
- Editor de nota de texto: toggle "🔒 Nota cifrada".
  - Guardar cifrada: cuerpo → Keychain; en SQLite `body = ''`, `secure = 1`. Tag manual permitido; **nunca** IA.
  - Abrir cifrada: `readSecureBody` dispara Face ID; si el usuario cancela, se muestra placeholder "Contenido bloqueado" con botón para reintentar.
  - Des-cifrar (toggle off): leer del Keychain (Face ID), escribir cuerpo en SQLite, borrar entrada del Keychain.
- Listado: badge 🔒, sin preview de cuerpo; búsqueda solo por título para estas notas.
- Borrar nota cifrada ⇒ `deleteSecureBody` además del row.
- Ocultas para: "Editar con voz", tags IA, share/preview con IA. El recordatorio solo expone el título (como hoy).
- Requisito de dispositivo: passcode configurado (si SecureStore falla por eso, mensaje claro).

## Testing

- Jest (lógica pura): `handleReminderAction` (fase 1), `isNoteTag` + parseo del nuevo JSON del formatter (fase 2), truncado/normalización de texto compartido (fase 3), validación de tamaño del cuerpo seguro (fase 5), migraciones sobre DB en memoria (fases 2 y 5, hay tests de repo existentes como patrón).
- Manual en iPhone (dev build): acciones de notificación con app abierta/cerrada, filtro por tags, share desde Safari/WhatsApp, "Oye Siri, dictar nota en Lumi", crear/leer/des-cifrar nota cifrada con Face ID y con Face ID cancelado.
- Fases 3-5 requieren nuevo prebuild + build del dev client; fases 1-2 corren sobre el build actual con solo JS.

## Fuera de alcance (esta versión)

- Control Widget / lock screen widget (fase futura).
- Guardado silencioso desde la share extension (sin abrir la app).
- Notas cifradas largas (AES) y cifrado de título.
- Tags libres o múltiples por nota.
